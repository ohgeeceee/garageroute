## Project overview

GarageRoute.com is a prototype web app for discovering garage/yard sales, previewing item inventory, and building an optimized driving route. It targets mobile-first users and includes seller-management features such as live inventory updates, a virtual queue, buyer messages, reservations/deposits, and donation-pickup requests.

The app is built with:

- **Framework:** Next.js 16 (App Router, Turbopack)
- **Language:** TypeScript 5 with strict mode
- **UI library:** React 19
- **Styling:** Tailwind CSS v4 (`@import "tailwindcss"` in `globals.css`)
- **Icons:** Lucide React
- **Maps:** Leaflet + OpenStreetMap (via `react-leaflet`)
- **Database ORM:** Prisma 6 with a SQLite file database
- **Mobile wrapper:** Capacitor 8 (iOS/Android packages installed, config in `capacitor.config.ts`)
- **Payments:** Stripe 22 integration for reservation deposits (falls back to mock mode if no Stripe key is configured)
- **PWA assets:** `public/manifest.json` and `public/sw.js`

This is a functional frontend/backend prototype. There is no user authentication system; seller access is gated by secret `sellerToken` URLs.

## Technology stack details

### Next.js / React

- Uses the **App Router** (`app/` directory).
- Server Components are the default; client components must include `"use client"`.
- Route handlers under `app/api/` use the `NextRequest` / `NextResponse` API.
- Dynamic route params are async (`params: Promise<{ id: string }>`).
- `next.config.ts` is currently empty; it relies on Next.js defaults.

### Database

- Prisma Client is instantiated as a singleton in `lib/prisma.ts`.
- SQLite database path comes from `DATABASE_URL` in `.env`.
- Schema models: `Sale`, `Item`, `Alert`, `Queue`, `Message`, `Reservation`.
- `photos` on `Sale` is stored as a JSON string and normalized to an array at runtime.

### Maps

- `DynamicMap` lazy-loads `MapView` on the client because Leaflet requires `window`.
- Default map center is Indianapolis, IN (`39.7684, -86.1581`) when no sales are present.

### Mobile / PWA

- Capacitor is configured with `appId: 'com.garageroute.app'` and expects a built `out/` directory.
- For a true Capacitor static build, `next.config.ts` will likely need `output: 'export'` and `distDir: 'out'`.
- `public/manifest.json` and `public/sw.js` provide basic PWA support; `ServiceWorkerRegistration.tsx` registers the worker.

## API surface

Public/data routes:

- `GET /api/sales?query=&type=&category=&verifiedOnly=` — list/filter sales
- `POST /api/sales` — create a sale (geocodes address via Nominatim)
- `GET /api/sales/[id]` — get sale details
- `POST /api/sales/[id]/queue` — join virtual queue
- `POST /api/sales/[id]/messages` — message seller
- `POST /api/sales/[id]/donate` — request donation pickup
- `POST /api/alerts` — sign up for alerts
- `POST /api/price-suggest` — get price suggestion for an item
- `POST /api/reservations` — create a reservation/deposit

Seller-management routes (token = `sellerToken`):

- `GET /api/manage/[token]` — load sale
- `PATCH /api/manage/[token]/items/[itemId]` — mark item sold/available
- `GET /api/manage/[token]/queue` + `PATCH /api/manage/[token]/queue/[queueId]`
- `GET /api/manage/[token]/messages`
- `GET /api/manage/[token]/reservations` + `PATCH /api/manage/[token]/reservations/[reservationId]`

## Code style guidelines

- **Language:** TypeScript with `strict: true`. Avoid `any`; ESLint is configured to flag explicit `any`.
- **Imports:** Prefer the `@/*` path alias (mapped to `./*`).
- **Components:** Write functional components. Add `"use client"` only when using browser APIs, React hooks, or client context.
- **Styling:** Use Tailwind utility classes. The design system uses `zinc-*` for neutrals, `blue-600` for primary actions, and `emerald-*` for success/impact accents.
- **Forms:** Native HTML inputs/selects with Tailwind styling; no external form library is used.
- **Linting:** ESLint uses `eslint-config-next/core-web-vitals` and `eslint-config-next/typescript` in a flat config (`eslint.config.mjs`).

## Testing instructions

There is **no test runner configured** in this project (no Jest, Vitest, Playwright, or Cypress).

The current quality gate is `npm run lint`. However, as noted below, lint currently fails.

## Environment variables

Create a `.env` file in the project root. Do not commit it. Required/used variables:

| Variable | Required? | Purpose |
|----------|-----------|---------|
| `DATABASE_URL` | Yes | SQLite DB path, e.g. `file:./prisma/dev.db` |
| `NEXT_PUBLIC_APP_URL` | No | Public base URL used for QR codes/fetch fallback; defaults to `http://localhost:3000` |
| `STRIPE_SECRET_KEY` | No | Enables real Stripe payments for reservations; without it, reservations run in mock mode |

## Security considerations

- **No authentication:** The only access control is the `sellerToken` UUID in `/manage/[token]` URLs. Treat these URLs as secrets.
- **Input validation:** API routes validate required fields and sanitize strings (`trim()`, `toLowerCase()`), but there is no rate limiting or CSRF protection.
- **Database:** Prisma queries are parameterized; no raw SQL is used.
- **Payments:** Stripe is only used if `STRIPE_SECRET_KEY` is present. In mock mode, a fake `paymentIntentId` is stored.
- **External images:** Sale/item photos currently use `picsum.photos` placeholders and are rendered with plain `<img>` tags. In production, migrate to `next/image` with a configured remote host list.
- **HTTPS:** Production deployments should enforce HTTPS for seller tokens and payment flows.
- **Geocoding:** Sale creation calls Nominatim with a custom `User-Agent`. Be mindful of usage policy and rate limits.

## Known issues / caveats

- `npm run lint` exits 0. A few `<img>` tag warnings remain; migrating to `next/image` with a remote host list is still recommended for production.
- The Capacitor config points to `out/`, but `next.config.ts` does not yet enable static export.
- There is no e-mail sending integration; alerts, queue notifications, and messages are stored in the DB only.
- The service worker caches dynamic API responses, which can become stale.

## Useful conventions

- Sale `photos` are stored as a JSON string in SQLite and normalized in API/client helpers.
- `impactKg` is calculated heuristically from item count (`items.length * 2.5` on seed, `* 1.5` on create).
- `RouteContext` holds the itinerary only in memory; it is not persisted.
- `/route` fetches driving directions from the public OSRM demo server and displays road-following polylines, total driving distance, and estimated duration.
- Default geographic fallback is Indianapolis, IN.
